---

## 一、系統定位(不可協商的邊界)

1. **單用戶**:只服務擁有者一人。無登入系統、無多用戶、無權限管理。
2. **Notion 為唯一真實資料源**:App 不儲存任何權威資料。所有讀寫經 Notion API 打回 Notion。允許記憶體/短期快取加速讀取,但快取永遠可丟棄、不可回寫衝突。
3. App 掛掉時,擁有者必須仍能直接在 Notion 手動完成一切操作——因此 App 不得建立「只有 App 看得懂」的資料格式。
4. 系統終點=Session 狀態「已交付」。排程與發布不在範圍,不做。
5. 訂閱制網站是另一個專案,本期不做任何面向客戶的功能。

## 二、技術原則

- 選主流、成熟、好維護的技術(建議:Next.js/React + 官方 Notion API SDK)。禁止新奇框架與重度依賴。
- 介面語言:繁體中文。行動裝置可用(擁有者常用手機)。
- 部署方式由擁有者另行決定(Vercel、Railway 等),交付物須含一鍵部署說明與環境變數清單(Notion token 由擁有者自行設定,不得寫死在程式碼)。
- **公網部署安全(v1.1)**:凡部署於公開網址,必須以環境變數 ACCESS_KEY 保護——所有頁面與寫入端點需驗證此金鑰(如首次進入輸入一次、存於瀏覽器)。這不是登入系統,是單一存取金鑰;無金鑰的請求一律拒絕。

## 四、頁面規格(對應 SOP 步驟)

### P1 主控台首頁
- 行事曆視圖:當月/下月切換,顯示所有任務節點(來源:Session 明細日期+序列展開結果+DB-15 場次)。實驗品任務加「實驗」視覺標記。
- 完成度儀表:依 DB-03 狀態機匯總百分比(已抽牌→已指定用途→解讀中→已產出→已交付),按當月/下月分視圖。
- 今日待辦區:今天到期的任務清單。

### P2 序列展開引擎(核心邏輯,SOP 第 3 步的自動化)
- 輸入:事件(DB-06)+日期(單日事件一個日期;多日事件由 DB-15 場次表取全部場次)。
- 處理:讀 DB-07 該事件的節點模板,依錨點計算實際日期——首場錨點相對第一場、每場錨點對每一場複製一組、末場錨點相對最後一場。
- 輸出:批次建立任務(寫入 DB-03/04 或任務清單),各掛計算出的日期。
- 滾動窗口規則:展開範圍=未來 35–40 天;例外:多日事件確認開課即全場次展開,不受窗口限制。
- 展開結果先以「預覽清單」呈現,**擁有者按確認才寫入 Notion**。

### P3 生產日工作台
- 照 SOP v0.9.2 的步驟順序做成引導流程(步驟清單+當前步驟高亮)。
- 第 2 步的支援:輸入月主題包關鍵字→自動列出 DB-11 匹配原子項清單(關聯比對 DB-16)。**只列清單,不自動組合**(A 方案定案)。
- 知識庫比對:以月主題標籤撈 DB-14 存貨清單供參考。
- **心理測驗維度/方法挑選為規則層動作**,由擁有者依 DB-14 對照表判斷,App **只負責撈清單供挑選**,不得實作自動配對演算法。

### P4 固定項目註冊器
- 表單:事件類型、事件型態(單日/多日)、宣傳期長度、序列節點編輯(相對天數、錨點、內容類型、目的、平台)。
- 送出→寫入 DB-06+DB-07。這是「新增固定項目=填表不改系統」的落實。

### P5 Session 工作站
- 批次建立:一鍵建 Session+N 筆明細(日期範圍自動展開,如 14 天占卜)。
- 抽牌輸入:選牌組→輸入牌號序列→自動組出識別碼(MP-15 格式)並驗證該牌存在於 DB-02。逆位以後綴 R 記於「抽出順序」字串(如 TA-05R),不新增欄位。
- 狀態機推進:單筆/批次改狀態,只允許順序推進(已抽牌→…→已交付),跳步需二次確認。

### P6 回饋快填(手機優先)
- 30 秒內完成:選對象(規則/Session)→四維評分(1–5 點選)→短評一行→送出。
- 問題類型標籤選擇時,顯示該標籤既有累積次數;達 3 次時提示「已達三次原則,可發起提案」。

### P7 提案審核台
- 待審提案清單(DB-10 狀態=待審)。
- 批准動作輔助:批准後引導完成規則升版(建立新版規則頁、狀態=現行、舊版改「舊版」、接上版本鏈)——每一步顯示給擁有者確認,不全自動。

### P8 生成工作台(v1.2 新增,v1.3 修訂,v1.4 同步;分兩階段)

**階段一・一鍵組稿(先做)**
選定一筆 Session → App 自動組裝完整提示詞 → 一鍵複製,擁有者貼入任何 AI 對話生成。零 API 成本。

提示詞組成(五項,缺一即報錯,不得靜默省略):
1. 該 Session 的牌卡資料(DB-03/04 → DB-02)
2. DB-05 對應規則之**現行版**(狀態=現行者)
3. 月主題包(DB-08)
4. **語氣指引現行版**(總綱 v2.5 定案:存於 **DB-14 一筆**,來源=原創,標題採「語氣指引 vX」;App 讀取版本最新者。**不為此新增 DB-05 欄位;若提議加欄位,一律駁回**)
   > 阻塞提醒:語氣樣本尚未到位前,此筆為空,P8 階段一無法通過驗收。
5. 該規則的輸出格式(DB-05「輸出格式」欄)

**階段二・一鍵生成(階段一實用驗證合格,且經擁有者明確批准後才開工)**

- 認證方式:**Claude Agent SDK + 擁有者訂閱之 Agent SDK 每月額度**。不使用獨立 API key。憑證由擁有者設定於環境變數,不得寫死。(依據:總綱 v2.5 第七章速查表「生成額度政策」)
- 事實依據(執行模型不得憑記憶推翻,如有疑義先查證再回報):
  - Agent SDK 官方文件載明,未經事先核准,不允許第三方開發者在產品中提供 claude.ai 登入或其速率限制。
    → 本 App 為**單用戶自用工具**,非對外產品,不提供任何登入功能,不在此限制情境內。
  - 自 2026-06-15 起,Pro/Max/Team/Enterprise 訂閱可領取**每月 Agent SDK 額度**,涵蓋 Agent SDK 使用、`claude -p`,以及基於 Agent SDK 建構的第三方應用;該用量**不計入訂閱一般用量上限**。
  - 參考:https://support.claude.com/en/articles/15036540-use-the-claude-agent-sdk-with-your-claude-plan
  - 開工前務必重新查證上述兩條是否仍為現況(政策會變),有出入即停下回報。
- **額度耗盡行為(v1.3 新增,採 A 方案)**:
  - 每月額度用完後,**不啟用 usage credits**。Agent SDK 請求將直接停止,直到次月額度重置。
  - App 必須:①顯示當月額度用量;②額度將盡時明確警示;③停止時明確報錯,不得靜默失敗或自動改用付費路徑。
  - 額度用盡時的正確行為=**退回階段一(一鍵組稿),手動貼入生成**。此即「內容不等系統」原則的落實,不斷更。
- 生成結果寫入草稿區,狀態=**待驗收**。
- 硬規則:
  1. 生成品**不得**自動標為已產出或已交付。
  2. 擁有者四維評分關卡不可跳過。
  3. 未來若開放對外服務(訂閱網站),**必須改用 API key**,不得沿用訂閱憑證。

## 六、Notion API 限制的處理要求

- 速率限制(約 3 請求/秒):批次操作必須排隊+節流,顯示進度條,不得硬打。
- 失敗重試:指數退避,3 次仍失敗→明確報錯並列出「哪幾筆沒寫入」,方便手動補。
- 部分成功的批次:必須告知成功/失敗清單,不得靜默吞掉。
- 多選/關聯值寫入前,一律以實際註冊選項清單**逐字比對**(異體字風險),不得憑字面相似判定;一筆非法值會使整批原子性失敗。

## 八、禁區

1. 不建登入、多用戶、通知推播。

4. 不做排程與發布整合。
5. 不引入需要額外付費或註冊的第三方服務(除部署平台)。**P8 階段二不啟用 usage credits;額度耗盡即停,不得自動轉入付費路徑**。

7. 遇未涵蓋情況:停下回報。
