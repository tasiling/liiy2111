# 開發送審流程(擁有者 2026-07-11 訂立)

## 規則:一個交付批次 = 一個 PR

批次的定義:對應委派書一次版號迭代(如 v1.5、v1.6),或委派書外的一次追加指示
(如 bug 修正、資料修正)。不得把下一批工作疊在上一批已合併的 PR 分支上——每次
開工前先確認該分支對應的 PR 狀態,若已合併,從最新 base 重新拉分支再動工。

## PR 標題

對應委派書版號或追加指示主題,例如:

- 「v1.6:P8-0 選項表單 + 明細狀態」
- 「Bug 修正:近期 Session 清單遺漏(建立日期脫鉤)」

## PR 描述必須包含兩件事

### ① 本批對應委派書哪些條目

列出本批對應《產線 C/A 委派書》哪一版、哪幾條,或是哪一次擁有者追加指示。

### ② 變更後三問

針對本批每一項牽涉資料欄位/狀態機的變更,逐項回答:

1. **存量資料在新邏輯下是什麼值?**(既有資料是否需要回填,或新邏輯本身就能安全處理舊值)
2. **新路徑建立的資料,每個必填欄位是否有值?**(新增/寫入路徑是否保證不留空)
3. **空值出現時系統行為是什麼?**(讀到空值時顯示/判斷怎麼處理,是否有防禦性 fallback,
   fallback 會不會誤寫回 Notion)

若某一項不適用(例如純讀取功能不寫入 Notion),需明確寫「不適用」並說明理由,不得省略不提。

## 部署與 PR 的關係

正式環境(Railway)只能從已合併的預設分支部署,不允許未送審的工作分支直接上線。
任何功能上線前必須先經過 PR 審查與合併。
