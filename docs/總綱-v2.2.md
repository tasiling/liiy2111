# 全零的深耕聚光系統 — 系統總綱 v2

> 文件性質:架構定案文件(凍結版)。本文件為所有分包委派書的共同附件。
> 執行原則:執行模型不得做架構決策。遇到本文件未涵蓋的情況,一律停下回報,不得自行判斷。
> 版本:v2.2|定稿日:2026-07-06|審核人:全零(系統擁有者,一切修改需其明確批准)

## 一、系統定位

- 單核心系統+多模組架構,非多部門並列。
- 底層邏輯為迴圈工程:輸入 → 判斷 → 執行 → 驗收 → 記錄 → 修正 → 下次更順。
- 系統目標:大幅解放更新壓力;穩定產出大眾占卜、心理測驗、邀請文、個人心得四條內容線;同時保留持續實驗與優化空間,且實驗不污染既有結構。
- 服務對象:月初能量流解讀、大眾占卜、日更內容、固定活動與報名內容、服務導流。

---

## 二、分層架構(污染防護的根基)

| 層級 | 內容 | 變動規則 |
|---|---|---|
| 事實層 | 神諭卡資料庫(牌組、編號、牌義、圖片) | 幾乎不動。修正模組**無權**修改此層 |
| 紀錄層 | 抽牌 session、回饋紀錄、素材庫 | 只增不改,歷史紀錄不可竄改 |
| 邏輯層 | 解讀規則庫、事件序列模板、輸出格式 | 僅透過提案審核制修改,全版本控制 |
| 排程層 | 主控台/任務行事曆 | 依模板自動展開,固定項目採註冊制 |
| 迴圈層 | 修正模組(守成)+精進區(開拓) | 雙迴圈,出口分別為規則升版與固定項目註冊 |
| 支援層 | 服務原子庫、知識庫、素材庫×2 | 各自獨立節奏,透過標籤與主系統掛鉤 |
| 未來層 | 訂閱制網站(抽牌、測驗遊戲、顯化工具等) | 暫不開發。所有資料結構須以「未來可被網站調用」為標準建置 |

**分層試錯原則**:試錯只發生在表現層(單次輸出),驗證過的才沉澱至邏輯層;事實層恆定。

## 三、模組清單與職責

### 1. 主控台(Dashboard)— 所有模組的入口
- 行事曆視圖:當月/下月切換。
- 任務完成度:依 session 狀態機匯總成百分比。
- **固定項目註冊器**:新增固定事件+其宣傳序列必須是「填表動作」,不是改系統。此為硬性需求。
- 實驗任務掛「實驗」標記,與固定項目視覺區分。

### 2. 神諭卡資料庫(事實層)
- 僅建神諭卡,不建塔羅、雷諾曼。
- 識別方式:牌組代碼+編號(例:MP-15,月相顯化牌第15張),避免跨牌組同號衝突。
- 欄位:牌組名稱/代碼、牌卡編號、牌名、關鍵字、短義、長義、圖片、使用場合標籤(能量流/大眾占卜/日更)。

### 3. 抽牌紀錄 Session(紀錄層)
- 抽牌事實與解讀邏輯分離:session 只記「抽到什麼」。
- **支援批次模式**:一筆 session 可掛多個日期、多組牌。
- 記錄「當時使用的規則版本號」。
- 狀態機:已抽牌 → 已指定用途 → 解讀中 → 已產出 → **已交付**(終點)。
- 排程與發布不歸本系統管(交付給排程機器人),發布後數據手動掛回。

### 4. 解讀輸入台
- 記錄三件事:項目(能量流/大眾占卜/日更/個案)+ 牌組 + 牌號。
- 不寫死規則;規則由規則庫決定。同一組牌可在不同規則下產出不同內容,不需重複建抽牌資料。

### 5. 解讀規則庫(邏輯層)
- 欄位:規則代碼、適用項目、牌位定義、解讀邏輯、輸出格式、**Canva 模板編號**(每種內容類型綁定固定模板,輸出即為可貼入模板的分頁結構)。
- 全版本控制:改動產生新版本(v1→v2),舊版不刪,可一鍵回退。
- 新月與滿月文案角度區分:新月=設定意圖、滿月=釋放清理,寫成兩條規則防同質化。

### 6. 任務行事曆(排程層,主控台的核心引擎)
- 定位:**批次生產排程系統**,非每日提醒器。
- **滾動窗口規則**:大生產日生產「未來 35–40 天內所有已知節點」,不以日曆月切齊(解決新月/滿月跨月序列斷裂問題)。
- 雙節奏:
  - 每月 1 個大生產日:能量流牌陣 → 月主題包 → 展開整月任務(二更、三更、所有事件序列文)。
  - 每雙週 1 個小生產日:批次抽一更大眾占卜 14 天份。
- **多日活動排程規則(v2.1 新增)**:
  - 事件分兩型:單日(新月、滿月、織光共作坊)與多日(共振會,如顯化、脈輪課)。
  - 多日活動一經確認開課,**全部場次立即進入行事曆,不受滾動窗口限制**;窗口僅約束內容生產的展開範圍。
  - 多日活動場次最先卡位,其他內容繞著排(它是唯一不可移動的承諾)。
  - **場次日三更照常(皆為預製),但不排實驗品**——開課日不斷更,是本系統的存在目的。
  - 場次節奏不設固定預設值(仍在測試中),每期開課時自訂。首個實例:2026-08 顯化主題,一期四天、每天 4 小時。

### 7. 修正模組(迴圈層・守成)
三層結構,定位為「審核閘門」而非生產部門:
- **回饋紀錄**:每筆回饋掛在具體對象(規則或 session)。來源=數據(觸及/互動/轉換)或個人評價。
- **三次原則**:同類問題出現 1 次記在 session;≥3 次才升級為規則層修正提案。防止單次波動亂改規則。
- **提案審核制**:提案含「改哪條規則哪個欄位、前後對照、依據哪些回饋、預期改善」。永遠為「待審」狀態,擁有者確認才生效。
- 個人評價四維度(各 1–5 分+短評):準確度/語感/轉換效果/可複用性。

### 8. 精進區(迴圈層・開拓)— 實驗室
- 職責:新格式實驗(文章架構、Reel 短片架構等),與修正模組(既有規則微調)分工。
- 欄位:實驗名稱、假設、格式描述、投放次數(建議 3–5 次)、數據+主觀復盤、狀態(實驗中/迭代中/畢業/淘汰)。
- **實驗佔用正式發布位**:不額外加更,保護產能上限,並使實驗與固定項目數據直接可比。
- 投放完畢自動觸發復盤任務出現在主控台。
- 復盤三選一:畢業(經固定項目註冊器轉正,取得自己的規則與序列模板)/迭代/淘汰(歸檔留紀錄)。

### 9. 服務原子庫(支援層)
- 服務項目細化為原子項,可組合搭配。
- 欄位:原子項名稱、**能量屬性標籤(可多選)**、價格、可組合性。
- 標籤詞彙表沿用能量流解讀常用詞(清理、顯化、關係、豐盛……),兩邊詞彙一致才能準確匹配。
- **A 方案(定案)**:系統只「撈料」——用當月能量關鍵字比對標籤,列出匹配原子項清單;三款主題服務由擁有者自行組合,填回月主題包。系統不做自動配菜。

### 10. 素材庫 ×2(支援層)
- **活動回饋箱**:所屬事件、日期、素材類型(截圖/文字/照片)、可否公開(含匿名處理規則)。餵事件序列的心得文。每次活動最低收 2–3 份可用回饋。
- **個人日記箱**:擁有者日記素材,餵個人心得產線。細則(日記如何進系統、AI 改寫尺度、不可公開清單)待實際使用後於精進區迭代。

### 11. 知識庫(支援層)
- **獨立產線+主題掛鉤(定案)**:有自己的生產節奏,不佔大生產日額度。
- 每篇內容打主題標籤;月主題包生成時,系統自動比對「本月主題相關存貨」,匹配者排入當月發布或於邀請文中引用。
- 定位:常青彈藥庫,平時累積、遇題複用。
- **進料來源三分類與加工紅線(v2.1 新增)**:
  - 原創:隨意使用。
  - 備課轉製(自己課程的備課資料,含 AI 製作的課堂檔案):隨意使用,並可反向導流(「這是共振會內容的冰山一角」)。可衍生:深度貼文、二更測驗題材、課前預習文(掛進場次序列)、課後複習包。
  - 學習筆記(他人課程、書籍):**必須內化改寫後才能公開**——只能輸出自己消化後的觀點與實踐經驗,不得輸出原課程的架構、講義、獨門術語。品牌權威建立在「全零的詮釋」上,不是轉述。
  - **所有知識轉製內容,發布或製作前一律經擁有者審核核可。**
  - 每筆知識庫素材記錄「衍生紀錄」,追蹤已被加工成什麼,避免重複。

## 四、內容生產結構(瀑布,非平行線)

```
月能量流牌陣(每月1次,主題源頭)
   ↓ 產出【月主題包】
   ├→ 三更・邀請文(結合當月三款主題服務,IG 為主)
   ├→ 二更・心理測驗(依賴月主題,月初批次設計 4–5 個輪用)
   ├→ 各事件宣傳序列文(新月/滿月/工作坊)
   └→ 知識庫存貨比對(相關者撈出複用)

一更・大眾占卜:雙週批次抽 14 天份(獨立於月主題)
個人心得產線:取自個人日記箱(獨立節奏)
```

**月主題包欄位**:主題名、能量關鍵字、當月三款主題服務(擁有者組合)、深度討論題目、每日互動方向。(不含視覺色調——已定案不加)

## 五、事件宣傳序列模板

**核心設計:心得文前置。** 心得文不是事件收尾,而是下一次同類事件的開場——活動後唯一動作是把回饋丟進素材庫(零寫作壓力),心得寫作發生在下次序列的預熱節點,取上次素材。

### 月能量流發布(每月 1 號)
| 節點 | 內容 |
|---|---|
| T+0 | 能量流解讀文+**當月所有固定活動總預告**(全月流量入口) |
| T+1~2 | 深度討論題目文(能量流延伸,導入互動) |

### 新月儀式(T-7 序列)/ 滿月儀式(T-7 序列,文案角度區分)
| 節點 | 內容 |
|---|---|
| T-7 | 預熱文=上次心得(取素材庫)+本次預告 |
| T-3 | 亮點文:本次儀式與月主題的連結 |
| T-1 | 最後提醒+準備事項 |
| T+0 | 當日限動/現場感內容 |
| T+1 | 回饋收集動作(丟素材庫,非發文) |

### 織光共作坊(T-14 序列,需兩波素材)
| 節點 | 內容 |
|---|---|
| T-14 | 開放報名文=上期學員心得+本期亮點 |
| T-10 | 價值文:這期解決什麼問題(結合服務品項) |
| T-7 | 案例/見證文(第二波素材) |
| T-3 | 名額倒數+常見 QA |
| T-1 | 最後召集 |
| T+0 | 當日內容 |
| T+1 | 回饋收集(丟素材庫) |

---

### 多日共振會(錨點制序列,v2.1 新增)
多日活動的節點不再只相對單一活動日,每個節點必須宣告**錨點**:
| 錨點 | 節點 | 內容 |
|---|---|---|
| 首場 | T-14 | 開放報名文=上期心得(素材庫)+本期亮點 |
| 首場 | T-10 | 價值文:這期解決什麼問題(結合服務品項) |
| 首場 | T-3 | 名額倒數+常見 QA |
| 每場 | 場前 1 日 | 該場提醒+預習指引(可掛知識庫課前預習文) |
| 每場 | 場後 | 回饋收集(丟素材庫,非發文) |
| 末場 | T+1 | 結業總回顧+心得收集(此批素材=下期 T-14 的預熱彈藥) |

「每場」節點隨場次數自動複製(4 場即生成 4 組)。場次節奏每期自訂,不設固定預設。

## 六、給執行模型的禁區清單

1. 神諭卡資料庫為事實層:**不得**增加欄位、修改編號規則或牌義內容。
2. 紀錄層只增不改:不得竄改歷史 session 與回饋紀錄。
3. 規則庫任何修改必須走提案審核制並產生新版本,不得覆蓋舊版。
4. 遇到本文件未涵蓋的情況:停下回報,不得自行做架構決策或「順手優化」。
5. 系統邊界:排程與發布(交付後)不在本系統範圍,不得擅自擴充。
6. 所有系統修改需擁有者明確批准後才可執行(提案→批准→執行)。
