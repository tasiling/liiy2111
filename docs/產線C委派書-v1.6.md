## 一、系統定位(不可協商的邊界)

1. **單用戶**:只服務擁有者一人。無登入系統、無多用戶、無權限管理。
2. **Notion 為唯一真實資料源**:App 不儲存任何權威資料。所有讀寫經 Notion API 打回 Notion。允許記憶體/短期快取加速讀取,但快取永遠可丟棄、不可回寫衝突。
3. App 掛掉時,擁有者必須仍能直接在 Notion 手動完成一切操作——因此 App 不得建立「只有 App 看得懂」的資料格式。
4. 系統終點=Session 狀態「已交付」。排程與發布不在範圍,不做。
5. 訂閱制網站是另一個專案,本期不做任何面向客戶的功能。

## 二、技術原則

- 選主流、成熟、好維護的技術(建議:Next.js/React + 官方 Notion API SDK)。禁止新奇框架與重度依賴。
- 介面語言:繁體中文。行動裝置可用(擁有者常用手機)。
- 部署方式由擁有者另行決定(Vercel、Railway 等),交付物須含一鍵部署說明與環境變數清單(Notion token 由擁有者自行設定,不得寫死在程式碼)。
- **公網部署安全**:凡部署於公開網址,必須以環境變數 ACCESS_KEY 保護——所有頁面與寫入端點需驗證此金鑰(如首次進入輸入一次、存於瀏覽器)。這不是登入系統,是單一存取金鑰;無金鑰的請求一律拒絕。

## 四、頁面規格(對應 SOP 步驟)

### P1 主控台首頁
- 行事曆視圖:當月/下月切換,顯示所有任務節點(來源:Session 明細日期+序列展開結果+DB-15 場次)。實驗品任務加「實驗」視覺標記。
- 完成度儀表:**批次 Session 以 DB-04 明細狀態匯總計算**(如「已產出 5/14」);單筆 Session 沿用 DB-03 狀態機。按當月/下月分視圖。
- 今日待辦區:今天到期的任務清單(以明細之對應日期為準)。

### P2 序列展開引擎(核心邏輯,SOP 第 3 步的自動化)
- 輸入:事件(DB-06)+日期(單日事件一個日期;多日事件由 DB-15 場次表取全部場次)。
- 處理:讀 DB-07 該事件的節點模板,依錨點計算實際日期——首場錨點相對第一場、每場錨點對每一場複製一組、末場錨點相對最後一場。
- 輸出:批次建立任務(寫入 DB-03/04 或任務清單),各掛計算出的日期。
- 滾動窗口規則:展開範圍=未來 35–40 天;例外:多日事件確認開課即全場次展開,不受窗口限制。
- 展開結果先以「預覽清單」呈現,**擁有者按確認才寫入 Notion**。

### P3 生產日工作台
- 照 SOP v0.9.2 的步驟順序做成引導流程(步驟清單+當前步驟高亮)。
- 第 2 步的支援:輸入月主題包關鍵字→自動列出 DB-11 匹配原子項清單(關聯比對 DB-16)。**只列清單,不自動組合**(A 方案定案)。
- 知識庫比對:以月主題標籤撈 DB-14 存貨清單供參考。
- 心理測驗維度/方法挑選為**規則層動作**,由擁有者依 DB-14 對照表判斷,App **只負責撈清單供挑選**,不得實作自動配對演算法。

### P4 固定項目註冊器
- 表單:事件類型、事件型態(單日/多日)、宣傳期長度、序列節點編輯(相對天數、錨點、內容類型、目的、平台)。
- 送出→寫入 DB-06+DB-07。這是「新增固定項目=填表不改系統」的落實。

### P5 Session 工作站
- 批次建立:一鍵建 Session+N 筆明細(日期範圍自動展開,如 14 天占卜)。**新建明細一律寫入明細狀態=待產出**。
- 抽牌輸入:選牌組→輸入牌號序列→自動組出識別碼(MP-15 格式)並驗證該牌存在於 DB-02。逆位以後綴 R 記於「抽出順序」字串,不新增欄位。
- **明細狀態操作(v1.6 新增)**:批次 Session 展開明細清單,支援**逐筆與多選**改明細狀態(待產出→已產出→已交付,只允許順序推進);表頭同步顯示匯總(如「已產出 5/14」)。全數明細=已交付時,提示擁有者將表頭狀態推進至「已交付」(一鍵,但需確認,不自動)。
- 表頭狀態機推進:單筆/批次改狀態,只允許順序推進(已抽牌→…→已交付),跳步需二次確認。

### P6 回饋快填(手機優先)
- 30 秒內完成:選對象(規則/Session)→四維評分(1–5 點選)→短評一行→送出。
- 問題類型標籤選擇時,顯示該標籤既有累積次數;達 3 次時提示「已達三次原則,可發起提案」。

### P7 提案審核台
- 待審提案清單(DB-10 狀態=待審)。
- 批准動作輔助:批准後引導完成規則升版(建立新版規則頁、狀態=現行、舊版改「舊版」、接上版本鏈)——每一步顯示給擁有者確認,不全自動。

### P8 生成工作台(分兩階段;v1.6 整併批次規格與組稿選項)

#### P8-0 組稿選項表單(v1.6 新增,兩階段共用的前置步驟)
組稿/生成前,擁有者先填本次的臨場選項:
1. **封面設定**:文字描述或 Canva 模板編號,擇一或並填;**選填**,留空時提示詞中省略封面區塊並明示「封面未指定」。
2. **維度/方法選擇**(僅心理測驗類任務出現):App 撈 DB-14 之 25 項對照表,列出候選服務(依當月能量關鍵字比對維度,多維度採命中任一),擁有者**點選**服務與方法(對照表方法為預設值,可換)。App 不代選、不自動配對。
3. 其他臨場備註(選填,一行,注入提示詞末端)。
表單內容注入提示詞,**不寫入任何資料庫**(臨場選擇非權威資料)。

#### 階段一・一鍵組稿(先做)
提示詞組成(五項,缺一即報錯並標明缺哪項):
1. 該 Session 的牌卡資料(DB-03/04 → DB-02)
2. DB-05 對應規則之**現行版**(狀態=現行者)
3. 月主題包(DB-08)。**例外**:一更大眾占卜獨立於月主題(總綱第四章),該類 Session 之月主題包為**選填**,缺之不報錯、於提示詞中省略該區塊。
4. **語氣指引現行版**:取 DB-14 標題以「語氣指引」開頭、核可狀態=已核可之最新版本筆;若其「內容」為指標說明且「連結」有值,**依連結取回全文**組入。不為此新增 DB-05 欄位;若提議加欄位,一律駁回。
5. 該規則的輸出格式(DB-05「輸出格式」欄)
加上 P8-0 表單內容。組稿結果先預覽,擁有者確認完整才提供複製。

**批次組稿的兩種模式**(App 依 Session 結構預判預設,擁有者可於組稿前手動切換;不得只實作其中一種):
- **模式甲・整批一稿**(全明細同規則之批次預設,典型:一更 14 天份):組出一份提示詞,內含全部明細(各明細=日期+牌卡),規則/月主題包/語氣指引/輸出格式各一份不重複;末尾自動加註「請依上列 N 個日期分別產出 N 篇,每篇獨立完整,依輸出格式分頁」。
- **模式乙・逐篇分稿**(明細各用不同規則、單篇長內容如能量流、或擁有者主動選擇時):每筆明細組一份獨立提示詞,以清單呈現,各附複製按鈕與日期/規則標示,可分次取用。
- 讀取 Notion 遵守速率限制:批次明細牌卡資料排隊+節流拉取,顯示進度。

  - Agent SDK 官方文件載明,未經事先核准,不允許第三方開發者在產品中提供 claude.ai 登入或其速率限制。本 App 為**單用戶自用工具**,非對外產品,不提供任何登入功能,不在此限制情境內。
  - 自 2026-06-15 起,Pro/Max/Team/Enterprise 訂閱可領取**每月 Agent SDK 額度**,涵蓋 Agent SDK、`claude -p` 及基於 Agent SDK 之第三方應用;該用量不計入訂閱一般用量上限。
  - 參考:https://support.claude.com/en/articles/15036540-use-the-claude-agent-sdk-with-your-claude-plan
- **批次生成一律採逐篇佇列**:每筆明細獨立一次生成呼叫,依序執行,顯示「第 N/M 篇」進度。不採整批一稿丟單次生成(長輸出品質漂移,且失敗整批重來)。每篇完成即寫入草稿區,狀態=**待驗收**,逐篇掛回對應明細。
- **額度守門(A 方案)**:
  - 佇列開始前顯示「本批預估 M 篇/當月餘量」;餘量明顯不足時警示,擁有者確認才開跑。
  - 佇列中途額度耗盡:停在當前篇,明確列出已完成/未完成清單,未完成者一鍵轉入階段一逐篇分稿(退回手動路徑)。不啟用 usage credits、不得靜默跳過、不得轉付費。

## 六、Notion API 限制的處理要求

- 速率限制(約 3 請求/秒):批次操作必須排隊+節流,顯示進度條,不得硬打。
- 失敗重試:指數退避,3 次仍失敗→明確報錯並列出「哪幾筆沒寫入」,方便手動補。
- 部分成功的批次:必須告知成功/失敗清單,不得靜默吞掉。
- 多選/關聯值寫入前,一律以實際註冊選項清單**逐字比對**(異體字風險),不得憑字面相似判定;一筆非法值會使整批原子性失敗。

## 八、禁區

1. 不建登入、多用戶、通知推播。
2. **AI 生成功能已解禁**,依第四章 P8 規格分兩階段實作;生成品一律為「待驗收」狀態,不得自動推進狀態機或跳過擁有者四維評分。內容**規則**仍屬產線 B 職權,App 只負責組裝與呼叫,不得內建任何規則邏輯或提示詞以外的創作判斷。
3. 不動 Notion schema:不新增欄位、不改選項、不建資料庫。
4. 不做排程與發布整合。
5. 不引入需要額外付費或註冊的第三方服務(除部署平台)。**P8 階段二不啟用 usage credits;額度耗盡即停,不得自動轉入付費路徑**。
6. **不實作維度/方法自動配對**——挑選權屬擁有者專業職權,App 只撈料、只提供點選。
7. P8-0 表單內容不寫入任何資料庫(臨場選擇非權威資料)。
8. 遇未涵蓋情況:停下回報。
