## 一、需求背景

P8 原規格為「選定一筆 Session → 組稿」。但本系統的主力生產型態是批次(一更 14 天份、三更整月份、二更 4–5 個),P8 必須直接支援批次,否則生產日仍需逐筆手動操作,違背系統存在目的。

## 二、P8 階段一・批次組稿(補完規格)

### 兩種組稿模式

**模式甲・整批一稿(同規則批次的預設)**
- 適用:一筆批次 Session 之全部明細共用同一條規則(典型:一更大眾占卜 14 天份)。
- 行為:組出**一份**提示詞,內含該 Session 全部明細(各明細=日期+牌卡資料),規則現行版、月主題包、語氣指引、輸出格式各一份,不重複。
- 提示詞末尾自動加註:「請依上列 N 個日期分別產出 N 篇,每篇獨立完整,依輸出格式分頁」。
- 理由:R-大眾占卜本就設計為批次規則(單篇結構固定),一稿多篇是它的原生用法。

**模式乙・逐篇分稿**
- 適用:①明細各用不同規則;②擁有者主動選擇(如整批一稿的產出品質不佳時);③單篇長內容(能量流解讀文)。
- 行為:每筆明細組一份獨立提示詞,以**清單**呈現,各附「複製」按鈕與該篇的日期/規則標示。
- 逐篇分稿不強迫一次全複製;擁有者可分次取用。

**模式選擇**:App 依 Session 結構預判(全明細同規則→預設甲;否則預設乙),但擁有者可在組稿前手動切換。**不得**只實作其中一種。

### 組稿共同規則(兩模式皆適用)
- 五項輸入(牌卡資料/規則現行版/月主題包/語氣指引/輸出格式)缺一即報錯,標明缺哪項。批次下的例外:一更大眾占卜獨立於月主題(總綱第四章),該類 Session 組稿時**月主題包為選填**,缺之不報錯、於提示詞中省略該區塊。
- 讀取 Notion 時遵守速率限制:批次明細的牌卡資料以排隊+節流方式拉取,顯示進度。
- 組稿結果先預覽,擁有者確認內容完整才提供複製。

## 三、P8 階段二・批次生成(補完規格)

- 批次生成一律採**逐篇佇列**:每筆明細獨立一次生成呼叫,依序執行,顯示「第 N/M 篇」進度。**不採整批一稿丟給單次生成**(單次長輸出品質漂移風險高,且失敗時整批重來)。
- 每篇生成完成即寫入草稿區,狀態=待驗收,**逐篇掛回對應明細**。
- 額度守門(A 方案,總綱 v2.5):
  - 佇列開始前顯示「本批預估 M 篇/當月餘量」;餘量明顯不足時警示,擁有者確認才開跑。
  - 佇列中途額度耗盡:**停在當前篇**,明確列出「已完成 1..N、未完成 N+1..M」,未完成者一鍵轉入階段一逐篇分稿(退回手動路徑)。不得靜默跳過、不得轉付費。
- 部分失敗處理:比照第六章 Notion 批次規則——指數退避重試 3 次,仍失敗者列入失敗清單,不得靜默吞掉。

## 四、驗收關卡(不因批次而變)

- 批次生成的每一篇**各自**為待驗收,**逐篇**過擁有者四維評分。不存在「整批一鍵通過」的按鈕或路徑。
- 驗收退回之單篇,可單獨重生成(帶原提示詞),不牽動同批其他篇。

## 五、本補丁驗收標準(追加)

- [ ] 一更 14 天批次 Session:模式甲組稿,提示詞含 14 組日期+牌卡且各項輸入不重複;月主題包缺席不報錯
- [ ] 同一 Session 手動切換模式乙,產出 14 份獨立提示詞清單,各可單獨複製
- [ ] 階段二批次:14 篇逐篇佇列生成,每篇獨立待驗收並掛回對應明細
- [ ] 模擬額度中途耗盡:正確停止、列出已完成/未完成清單、未完成者可轉階段一;無任何付費請求
- [ ] 單篇退回重生成,不影響同批其他篇狀態

## 六、邊界(不變)

- 本補丁未提及之 P8 規格,一律以委派書 v1.4 為準。
- 硬規則不因批次鬆動:生成品不得自動推進狀態機、不得跳過四維評分、額度耗盡不轉付費。
- 遇未涵蓋情況:停下回報。
