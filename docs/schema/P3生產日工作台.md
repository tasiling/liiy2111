# P3 生產日工作台(委派書 §四,第二期)

## 開工前 schema 核對

- DB-08 月主題包:「能量關鍵字」是關聯 DB-16 標籤詞庫的 relation(可多筆),先前
  `mapMonthlyTheme()` 未讀取此欄位,本次補上。
- DB-11 服務原子庫:「原子項名稱」(title)、「狀態」(單選:啟用/停用)、「能量屬性
  標籤」(關聯 DB-16,可多筆)、「可組合性」(text)、「價格」(number)。
- DB-14 知識庫:除既有的標題/內容/來源/核可狀態/連結外,還有「主題標籤」(關聯 DB-16,
  可多筆)與獨立的「狀態」單選欄(存貨/已排入/已發布)——**注意這個「狀態」跟「核可
  狀態」是兩個不同欄位**,P3 比對只看「狀態」,不篩「核可狀態」。

## 比對規則(擁有者 2026-07-12 裁決,委派書「命中任一」定案的具體化)

1. **匹配規則**:月主題包「能量關鍵字」與 DB-11「能量屬性標籤」/DB-14「主題標籤」
   只要共享至少一個標籤(交集≥1)就列入,不要求全部命中。
2. **DB-11 只列狀態=啟用**:停用的原子項不出現在清單中(呼叫端先過濾,見
   `listActiveServiceAtoms()`)。
3. **排序**:命中標籤數多者在前,同數依原子項名稱/標題排序。只列清單附命中標籤,
   不加推薦分數、不自動組合三款主題服務(A 方案,擁有者親自組合)。
4. **月主題包未設關鍵字**:明確提示「本月能量關鍵字未設定」,不 fallback 列出全部
   原子項/知識庫條目(避免誤導成「這些都跟本月有關」)。
5. **DB-11 空表**:目前為已知狀態(建檔工作單在擁有者手上),不擋 P3 開工——顯示
   「原子庫尚未建檔」,不是報錯;與「有資料但沒比對到」(顯示「本月無匹配原子項」)
   分開呈現,不要混成同一句空白提示。
6. **DB-14 比對同規則,狀態=存貨者置頂**:交集≥1 即列入(不篩核可狀態,不排除
   已排入/已發布,只是排序上把「狀態=存貨」的排到最前面),同一層內再依命中標籤數
   →標題排序(`matchKnowledgeEntries()`)。

比對純函式在 `lib/match.ts`(`matchServiceAtoms`/`matchKnowledgeEntries`),不碰
Notion,只做記憶體內比對,方便獨立驗證與未來調整規則。API 路由
`GET /api/production-day/matches?month=YYYY-MM` 組裝資料並附上命中標籤名稱
(查 DB-16 標籤詞庫解析 id→名稱)。

## 引導流程範圍(委派書:「照 SOP 步驟順序做成引導流程,步驟清單+當前步驟高亮」)

`app/production-day/page.tsx` 依《大生產日 SOP v0.9.2》主流程第 0–5 步做成靜態步驟
清單(標題、預估時間、SOP 原文摘要、相關頁面連結),支援:

- 點擊步驟設為「目前步驟」(外框高亮)。
- 每步可勾選「已完成」(打勾+刪除線),純前端 UI 狀態。
- 第 2 步嵌入 DB-11 匹配清單、第 3 步嵌入 DB-14 比對清單。
- 第 3 步的「二更測驗維度/方法挑選」不重新實作——委派書明訂 App 只負責撈清單供挑選、
  不得自動配對,這個功能已經是 P8-0 組稿選項表單的一部分(`/generate` 頁面),第 3 步
  說明文字附連結過去,不重複建置。

**目前第幾步、哪幾步已勾選完成,是純前端 state,重新整理頁面會重置,不寫入
Notion。** 理由:委派書系統定位第 2 條「App 不得建立『只有 App 看得懂』的資料格式」
與「Notion 為唯一真實資料源」——生產日的步驟進度不是需要跨裝置同步的權威資料,而是
單次引導用的輔助狀態,比照這個定位不新增 Notion 欄位來存它。若擁有者需要跨裝置或
跨次的進度保存,需另外討論存放位置(不能是新的 DB-06/07 之類的權威欄位)。
