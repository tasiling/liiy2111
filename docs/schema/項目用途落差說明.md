# DB-03 項目用途 欄位落差說明(v2:已定案)

> 這不是規則修改,是產線 C 執行過程中發現的 schema 落差記錄。v1 提出的「留空+備註」方案已被
> 擁有者否決(結構化資訊放進自由文字違反總綱總則),改採方案為正式定案,記入產線 A 委派書 v1.3。

## 問題(v1 記錄)

DB-03 抽牌 Session 的「項目用途」select 欄位,原本 schema 選項只有 5 個:

- 能量流、大眾占卜、日更、個案、實驗

但 P2 序列展開引擎(SOP 第 3 步)產出的任務(新月/滿月/共振會的預熱文、亮點文、名額倒數、
回饋收集等)、以及三更邀請文、二更心理測驗,都不屬於這 5 個選項,對不上分類。

## 定案做法(v1.3)

擁有者已在 Notion 手動於 DB-03「項目用途」新增 4 個選項:**事件序列文、邀請文、心理測驗、
個人心得**(App 不得代勞新增選項,此為擁有者親自操作)。對應規則:

| 內容類型 | 項目用途 |
|---|---|
| P2 序列展開引擎產出的任務(讀 DB-07 序列節點表) | 事件序列文 |
| 三更邀請文 | 邀請文 |
| 二更心理測驗 | 心理測驗 |
| 個人心得產線(取自個人日記箱) | 個人心得 |

程式碼對應:`lib/notion/schema.ts` 的 `SESSION_項目用途`(完整 9 選項清單)、
`EXPAND_ENGINE_項目用途`(=事件序列文,P2 commit 路由已使用)、`三更_項目用途`、
`二更_項目用途`(=邀請文/心理測驗,供未來實作三更/二更產生功能時使用)。

**目前僅 P2 序列展開引擎已上線並使用「事件序列文」寫入。三更邀請文、二更心理測驗的
自動產生功能不在第一期 MVP(P1/P5/P2)範圍內,尚未實作——上面兩個常數是預留給那支功能
未來實作時使用,現在程式碼沒有任何地方呼叫到它們。**

## 仍未解決的次要落差:DB-04 明細缺乏節點/內容類型欄位

DB-04 抽牌明細本身沒有欄位可記錄「這筆對應 DB-07 哪個節點/哪種內容類型」(只有
明細編號、對應日期、所屬 Session、抽出牌卡、抽出順序)。目前做法:P2 commit 時把節點
資訊編碼進「明細編號」(title,自由文字),例如 `顯化-202608-01_T-14預熱文`,不是
結構化欄位,查詢/統計較不方便。若擁有者未來想要更精確,可循同樣的提案審核制解決
(例如比照這次,手動幫 DB-04 加一個文字欄位)。
