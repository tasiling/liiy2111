# DB-03 項目用途 欄位落差說明

> 這不是規則修改,是產線 C 執行過程中發現的 schema 落差記錄,供日後提案參考。

## 問題

DB-03 抽牌 Session 的「項目用途」select 欄位,實際 schema 選項為:

- 能量流、大眾占卜、日更、個案、實驗(共 5 個)

但 P2 序列展開引擎(SOP 第 3 步)產出的任務,內容類型包括:新月/滿月/共振會的預熱文、
亮點文、名額倒數、回饋收集,以及三更邀請文、二更心理測驗——這些都不屬於上述 5 個選項。
對照組 DB-05 解讀規則庫的「適用項目」欄位反而有 6 個選項(能量流、大眾占卜、心理測驗、
邀請文、個人心得、事件序列文),兩邊選項集不一致。

## 目前做法(執行者判斷,未經擁有者逐項確認,可隨時覆蓋)

1. 序列展開引擎寫入 DB-03 時,「項目用途」留空,不硬塞不精確的選項,也不新增選項
   (委派書禁區明訂 App 不得改 select 選項)。
2. Session「備註」欄記錄本批次來源事件與節點摘要。
3. DB-04 明細沒有欄位可記錄「這筆對應哪個節點/內容類型」,因此把節點內容類型編碼進
   「明細編號」(title,自由文字),例如:`顯化-202608-01_T-14預熱文`。

## 待決:如果擁有者希望更精確分類

- 選項一:擁有者在 Notion 手動幫 DB-03 項目用途新增選項(如「事件序列文」「邀請文」
  「心理測驗」「個人心得」),App 再改一行程式碼對應寫入。
- 選項二:讓 DB-03 項目用途 與 DB-05 適用項目 兩邊選項集統一,避免未來新規則類型
  又對不上 Session 分類。
- 選項三:接受現狀(留空 + 備註/標題編碼),不特別處理。

此檔案本身不是規則,若要正式定案,請走總綱的提案審核制。
