# P6 回饋快填(委派書 §四,第二期,擁有者指示 P6 優先於 P3)

## 開工前 schema 核對

- DB-09 回饋紀錄:重新 fetch 確認欄位——「對象類型」單選(規則/Session)、「對象-Session」
  「對象-規則」各自對應的 relation(依對象類型擇一填)、「來源」單選(數據/個人評價)、
  「準確度/語感/轉換效果/可複用性」四個 number 欄(description 皆註明「1–5,個人評價時填」)、
  「問題類型標籤」、「短評」、「日期」。
- **重要發現**:「問題類型標籤」是 DB-09 自身的 **multi_select** 欄位(目前僅一個選項
  「語感偏硬」),**不是**關聯到 DB-16 標籤詞庫的 relation。DB-16 目前只接了月主題包/
  服務原子/知識庫/素材四個資料庫,沒有接回饋紀錄。因此「累積次數」的統計邏輯是直接
  掃描 DB-09 全部記錄、逐筆計算各選項出現次數,不需要查 DB-16。
- 目前 Notion 端問題類型標籤只有「語感偏硬」一項,`lib/notion/schema.ts` 的
  `FEEDBACK_QUESTION_TAGS` 常數對照此清單;日後擁有者於 Notion 新增選項後,需重新
  fetch schema 更新此常數,App 不得自行新增選項(比照多選欄位一貫的「逐字比對」規則,
  這裡用前端固定選項清單的方式落實,而非另外寫執行期驗證)。

## 設計決策(委派書未明訂,執行時的合理預設,已經擁有者 2026-07-12 查核回覆裁決)

1. **「來源」欄自動寫入「個人評價」**——**照准**。委派書 P6 規格描述的四維評分操作
   (擁有者手動 1–5 點選)與 DB-09「來源=個人評價時填」的欄位說明完全對應,建立時
   直接寫入,不另外讓擁有者選(減少操作步驟,符合「30 秒內完成」的要求)。若擁有者
   之後有「數據」來源的回饋要記錄(如自動化分析產生),那不是 P6 這個手動快填表單的
   範圍,需另外處理。
2. **回饋編號格式** `FB-YYYYMMDD-流水號`——**照准**,比照 Session 編號慣例
   (`nextFeedbackCode()`,邏輯與 `nextSessionCode()` 同構)。
3. **「對象-規則」選單改為兩段式**——**擁有者裁決調整**:原提案是「列出全部現行版規則,
   不分適用項目」,擁有者回覆改為兩段式呈現——**最近 14 天內有 Session 使用過的現行
   規則置頂**,其餘收進「更多」展開。純前端過濾顯示順序,不動 Notion 資料、不改變
   `listCurrentRules()` 回傳的規則清單本身。
   - 實作:`listRecentlyUsedRuleIds(days)` 查最近 N 天建立的 Session(沿用
     `listSessionsCreatedBetween`),取其 DB-03「使用規則」關聯欄的聯集,得到「最近
     用過」的規則 id 清單。`GET /api/rules` 同時回傳 `rules`(全部現行版)與
     `recentlyUsedIds`(最近 14 天用過的 id 集合)。
   - 前端 `RulePicker`:`recentlyUsedIds` 命中的規則置頂顯示;其餘收在「更多規則(N)」
     摺疊區,點擊展開。若最近 14 天沒有任何 Session 用過現行規則(置頂清單為空),
     直接顯示全部規則並提示「最近 14 天沒有 Session 用過現行規則,以下為全部規則」,
     不呈現空的置頂區塊。
   - 「對象-規則」的挑選範圍仍是「不分適用項目」,這條原判斷未變——只是原本的單一
     落落長清單改成分兩段呈現,降低常用規則被埋在長清單裡的操作成本。
4. **「日期」自動=送出當下**——未在查核清單中提及,維持原判斷:不提供日期選擇器,
   固定寫入伺服器當下日期,符合快填場景(即時記錄當下的回饋,不是回溯登記)。
5. **「現象描述」欄位未在 P6 表單提供**——未在查核清單中提及,維持原判斷:委派書
   規格明確列出「短評一行」,未提到「現象描述」;現象描述留空不影響任何狀態機或
   必填邏輯(純文字選填欄位),日後若擁有者需要更詳細的描述輸入,再議是否加欄位。

## 三次原則提示

`GET /api/feedback/tag-counts` 回傳目前每個問題類型標籤在 DB-09 全部記錄中出現的既有
次數。P6 表單勾選標籤時,選項旁即顯示「既有 N 次」;累積次數達 `FEEDBACK_PROPOSAL_THRESHOLD`
(=3)即顯示「已達三次原則,可發起提案」提示。此為純顯示提示,**不會**自動建立 DB-10
提案(P7 提案審核台屬第三期範圍,尚未實作;發起提案仍是擁有者自行操作)。

## 寫入驗證落實

- 四維評分 1–5 範圍:`validateFeedbackScore()`,App 端強制檢查後才寫入 Notion(委派書
  五之 1;Notion number 欄位本身無法強制範圍)。
- 問題類型標籤:僅能從 `FEEDBACK_QUESTION_TAGS` 固定清單勾選,不接受任意文字,避免
  寫入未註冊的選項值。
