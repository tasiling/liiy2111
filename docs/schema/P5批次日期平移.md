# P5 批次日期平移(擁有者 2026-07-13 追加指示)

## 規格對照

1. 批次 Session 展開明細後提供「整批平移日期」:輸入新起始日或 ±N 天,App 計算全部
   明細新日期,保持原本間隔結構。
2. 平移前顯示預覽(明細編號|原日期→新日期),擁有者確認才寫入。
3. 寫入遵守速率限制排隊,部分失敗列清單。
4. 只平移明細狀態=待產出的明細;已產出/已交付不可動,混批時只平移待產出部分並
   明確提示跳過哪幾筆。
5. 平移完成後自動在 Session 表頭備註追加一行異動紀錄。
6. 單筆明細日期編輯同樣開放,同樣限待產出、同樣留軌跡於表頭備註。

## 實作方式

**用「整批加同一個天數」達成保持間隔結構**,不是逐筆重新計算——不論明細是逐日連續
(如 14 天占卜)或未來可能出現的非連續批次,對每筆明細的「對應日期」統一加上同一個
`offsetDays`,彼此間的相對天數自動不變,不需要額外的「間隔保持」邏輯。

- **輸入方式**:前端提供兩種模式互斥切換——「指定新起始日」或「位移天數(±N)」。
  選「指定新起始日」時,`offsetDays` = 新起始日 − 待平移明細中最早的「對應日期」
  (只算明細狀態=待產出者,已產出/已交付的日期不列入這個「原起始日」的計算,因為
  它們本來就不會被移動)。兩種模式最終都化為同一個 `offsetDays` 送給後端,後端只認
  這一個參數,不重複兩套邏輯。
- **預覽**:前端用已載入的明細清單(`details`)在瀏覽器端直接算,不另開一支預覽 API——
  按「整批平移日期」展開的面板即時顯示每筆「原日期→新日期」,已產出/已交付的明細
  也列出但標示「不變——已產出,跳過」,讓擁有者一眼看到哪些會動、哪些不會動。
- **寫入**:`POST /api/sessions/[id]/shift-dates`,body 只有 `offsetDays`。伺服器端重新
  查一次明細(不信任前端算好的清單,避免預覽後、確認前資料被別處改動的競態),依
  明細狀態分成可平移/跳過兩組,`runBatch` 逐筆節流寫入,回傳
  `{succeeded, failed, skipped}` 三個清單,不靜默吞掉任何一種結果。
- **表頭備註軌跡**:寫入成功至少一筆後,呼叫 `appendSessionNote()` 在 Session「備註」
  欄追加一行,格式:
  ```
  【日期平移】2026-07-13~2026-07-26 → 2026-07-20~2026-08-02(+7 天,操作日期 2026-07-13,跳過 1 筆已產出/已交付明細)
  ```
  `appendSessionNote()` 是「讀現有備註→接一行→整段寫回」,不覆蓋既有內容;`mapSession()`
  因此新增讀取「備註」欄(先前只有寫入沒有讀取)。
- **單筆日期編輯**:`PATCH /api/details/[id]/date`,伺服器端驗證明細狀態=待產出才放行
  (非待產出回 409,訊息明講「已產出/已交付明細不可更改日期,紀錄層只增不改」),
  成功後查該明細的「所屬 Session」,同樣追加一行備註(格式:
  `【日期修改】明細 XXX-05:2026-07-05 → 2026-07-12(操作日期 2026-07-13)`)。
- **P5 前端**:明細列的日期從純文字改成 `DetailDateEditor`——明細狀態=待產出時顯示
  「編輯」連結可改日期,非待產出時顯示「(已鎖定)」,不給編輯入口(不是隱藏改期
  按鈕後端才擋,而是前後端都擋,UI 上先講清楚為什麼不能改)。

## 變更後三問

1. **存量資料**:此功能不改變既有明細「對應日期」的值本身,只在擁有者主動操作時
   才寫入;既有 Session 若「備註」欄本來是空的,`appendSessionNote()` 判斷
   `existing` 為空字串時直接寫入這一行,不會出現開頭多一個換行的空行。
2. **新路徑必填欄位**:此功能不建立新資料,只更新既有明細/Session 的既有欄位
   (對應日期、備註),不適用「新建資料必填欄位」的問題。
3. **空值行為**:
   - 批次平移時,若該批次沒有任何明細狀態=待產出(全部已產出/已交付),後端回
     400「沒有可平移的明細」,前端也會先行提示同樣訊息,不會送出空的平移請求。
   - 明細若缺少「對應日期」(理論上不會發生,建立時必填),平移時直接跳過並歸類
     為「不可平移」,不會因為 `null` 日期讓計算出錯。
   - 單筆日期編輯時,若目標明細非待產出,後端回 409 並附清楚原因,不是泛用的
     500 錯誤。
