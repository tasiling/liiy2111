# 明細狀態與完成度儀表(委派書 v1.6)

## 明細狀態機

DB-04 抽牌明細新增「明細狀態」單選欄:待產出 → 已產出 → 已交付,順序固定,終點=已交付。
與 DB-03 表頭「狀態」是兩套獨立的狀態機,互不覆蓋:

- 表頭「狀態」(已抽牌→已指定用途→解讀中→已產出→已交付):Session 整體生命週期,單筆模式主要用這個。
- 明細「明細狀態」(待產出→已產出→已交付):批次 Session 底下每一天的實際生產進度。

## 寫入驗證(委派書 v1.6 五、之 10、11)

10. **新建明細一律待產出**:`lib/notion/mutations.ts` 的 `createDetail()`/`createDetailWithTitle()`
    建立時一律寫入 `明細狀態=待產出`,不留空、不需要呼叫端額外指定。
    **明細狀態只允許順序推進**:`canAdvanceDetailStatus()` 檢查 next 必須恰為 current 的下一階,
    不像表頭狀態機那樣開放「跳步+二次確認」——明細只有三階,不開放任意跳轉。
11. **表頭「已交付」推進提示僅在全數明細已交付時出現,且需擁有者確認**:P5 頁面
    (`app/sessions/page.tsx`)在展開的明細清單全部等於「已交付」且表頭狀態尚未是「已交付」時,
    才顯示確認按鈕;按下後才呼叫既有的表頭狀態推進 API,不會自動觸發。

### 空值防禦(擁有者 2026-07-11 追加指示)

補齊「明細狀態」欄位前建立的 17 筆舊明細,擁有者已在 Notion 手動回填。為避免日後任何管道
(手動建立、匯入、未來新功能忘了設定)再產生空值而讓儀表出現「黑數」,`lib/notion/schema.ts`
新增 `normalizeDetailStatus()`:讀到空值或非法值一律視同「待產出」。所有讀取「明細狀態」做
判斷或顯示的地方(P1 完成度儀表統計、P5 表頭匯總、P5 逐筆/多選推進的目前狀態判定、P5 明細
狀態顯示)都套用這個 fallback。**寫入時仍照常寫入使用者選的真值**,不會把 fallback 值寫回
Notion——下一次推進動作發生時,那筆明細就會有真正的狀態,不需要額外的一次性回填腳本。

### 表頭狀態空值防禦 + 建立日期與內容日期脫鉤(擁有者 2026-07-11 追加指示,bug 追蹤)

擁有者回報:S-20260713-001 與 S-20260706-01 在 DB-03 的狀態/模式/項目用途完全一致,但 P5
「近期 Session」清單只顯示後者,重新整理無效。查核結果:

- **不是快取問題**:全專案(`lib/`、`app/`)無任何快取邏輯,也無 Next.js 快取指令
  (`dynamic`/`revalidate`/`force-static` 等),每次請求都直接打 Notion API。
- **根因**:`/api/sessions` 的「近期 Session」查詢(`listSessionsCreatedBetween`)以
  DB-03「建立日期」篩選,區間為「今天往前 60 天〜今天」,不排序、不額外限制筆數。
  但批次/單筆建立 Session 時,原本把「建立日期」直接寫成使用者選的內容起始日
  (`startDate`/`date`)——若該批次是提前規劃未來日期(如 S-20260713-001 選了
  2026-07-13),建立日期就會被寫成未來,超出查詢區間上界(今天),清單自然看不到它,
  且與快取無關,重新整理不會恢復。

修正:`app/api/sessions/batch/route.ts`、`app/api/sessions/single/route.ts` 建立 Session 時,
「建立日期」一律寫入伺服器當下日期(`new Date().toISOString().slice(0,10)`),與內容日期脫鉤
——內容日期本就獨立記在每筆明細的「對應日期」,不受影響。既有的 S-20260713-001 一筆已直接於
Notion 修正「建立日期」為其實際建立當天(2026-07-08)。

另外比照 `normalizeDetailStatus`,`lib/notion/schema.ts` 新增 `normalizeSessionStatus`:
DB-03「狀態」讀到空值/非法值一律視同狀態機起點「已抽牌」,套用於狀態推進 API
(`app/api/sessions/[id]/status/route.ts`)、P5 清單狀態顯示與推進下拉、P1 完成度儀表單筆
計數(`app/api/dashboard/route.ts`)。Session 建立時本就強制寫入狀態(`createSession`),
此 fallback 是額外一層防線,不是取代寫入時的預設值。

## P5 明細狀態操作

- **逐筆**:每筆明細列旁邊有「推進明細狀態」下拉,只列出目前狀態之後的合法選項。
- **多選批次**:勾選多筆明細後,若所選明細目前狀態**一致**,才會出現「推進為下一階」按鈕
  (按鈕文字直接顯示下一階名稱,不是任選目標狀態,避免對混合狀態的選取產生跳步歧義)。
  若所選明細狀態不一致,畫面會提示「請個別操作或篩選同狀態再批次」,不猜測、不強行套用。
  批次更新走 `POST /api/details/batch-status`,逐筆各自驗證與寫入,失敗的那幾筆列入失敗清單
  並附原因,不影響其餘筆數(委派書六、Notion API 限制處理要求)。
- **表頭匯總**:明細清單上方顯示「待產出 X/N、已產出 Y/N、已交付 Z/N」,即時反映目前的批次進度。

## P1 完成度儀表

`GET /api/dashboard` 的 `completion` 改為兩組獨立統計,不再混用同一組 5 階狀態:

```
completion: {
  total, done,                          // 合併後的總數/已完成數,供整體進度條使用
  single: { total, byStatus },          // 單筆 Session,依 DB-03 狀態機(5 階)
  batch:  { total, byStatus },          // 批次 Session 的明細,依 DB-04 明細狀態(3 階)
}
```

判斷依據:明細所屬 Session 的「模式」欄位。模式=批次者,以其在本月「對應日期」範圍內的明細,
逐筆計入 `batch` 統計;模式=單筆者,以 Session 本身的「狀態」計入 `single` 統計(每個單筆
Session 只計一次,不因擁有多筆明細而重複計入——雖然單筆 Session 定義上只有一筆明細)。
整體進度條的完成度 = (single 已交付數 + batch 已交付數) / (single 總數 + batch 總數)。

## P8-0 組稿選項表單

委派書 v1.6 第四章新增,純前端注入提示詞,不寫入任何資料庫:

- **封面**:文字描述、Canva 模板編號皆選填,擇一或並填。兩者皆空時提示詞的封面區塊
  明示「封面未指定」,不是省略不寫(避免使用者以為系統忘了組這段)。
- **維度/方法**(僅 Session 項目用途=心理測驗時顯示):候選清單來自 DB-14
  「核可狀態=已核可」的知識庫條目,解析其「內容」欄的固定文字格式
  (`主維度:...`/`方法:...`/`方法說明:...`/`題目1-3:...`,見
  `lib/generate/serviceDivination.ts`)。**只列清單供點選,不做任何維度/方法自動配對**
  (委派書禁區第 6 條)。方法欄位預設為候選項解析出的值,但提供七方法封閉清單下拉可換。
- **其他臨場備註**:一行選填文字,附加在提示詞末端。

三者組成的區塊由 `lib/generate/p8zero.ts` 的 `buildP8ZeroSection()` 統一格式化,
單篇組稿(`compose.ts`)與批次組稿(`app/generate/page.tsx` 前端組裝)共用同一份邏輯。
