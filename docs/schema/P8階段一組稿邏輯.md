# P8 階段一・一鍵組稿 邏輯說明

對應委派書 v1.4 第四章 P8。純讀取 + 文字組裝,不呼叫任何 AI、不寫入 Notion,零 API 成本。

## 五項輸入的來源與比對邏輯

| 項目 | 來源 | 比對方式 |
|---|---|---|
| 1. 牌卡資料 | DB-03/04 → DB-02 | 取 Session 指定明細的「抽出順序」字串(權威順序,非關聯欄位),逐一解析後查 DB-02 |
| 2. 對應規則現行版 | DB-05 | 用 Session 的「項目用途」比對 DB-05「適用項目」,篩「狀態=現行」 |
| 3. 月主題包 | DB-08 | 用使用者指定的月份(YYYY-MM)直接比對 DB-08「月份」 |
| 4. 語氣指引現行版 | DB-14 | 篩「來源=原創」且「標題」以「語氣指引」開頭,解析標題內的版本號 vX,取最大值 |
| 5. 輸出格式 | DB-05(同第 2 項規則頁的「輸出格式」欄) | 與第 2 項同一次查詢取得,不必再查一次 |

## 明細挑選邏輯(對應 SOP「逐日生成」)

- Session 模式=單筆:只有一筆明細,自動取用,不需使用者指定。
- Session 模式=批次(如 14 天占卜批次):**一次組稿只對應一天**,使用者必須在頁面上選定其中一筆明細(對應 SOP 小生產日「套規則逐日生成」——批次建立是一次性的,但生成是逐日的)。未指定則報錯,不會自動挑第一筆或全部塞在一起。

## 逆位記法解析

總綱 v2.3 定案:逆位不新增欄位,以「抽出順序」字串內的識別碼後綴 `R` 表示(如 `TA-05R`)。
`lib/generate/cards.ts` 的 `parseDrawOrder()` 依此解析每個識別碼是否逆位,再用不含 R 的識別碼查 DB-02。

## 語氣指引版本挑選

`lib/generate/toneGuide.ts` 的 `pickLatestToneGuide()`:對候選清單逐一用正規表達式
`/語氣指引\s*v(\d+)/i` 抓版本號,取數字最大者。標題格式不符(抓不到版本號)的候選會被忽略,
不會誤判為最新版。

## 缺一即報錯

`lib/generate/compose.ts` 的 `composePrompt()` 五項來源分別檢查,任何一項缺漏都會累積進
`missing: string[]` 一次性回報(不是查到第一個缺漏就中斷),讓使用者一次看到所有卡住的地方,
不需要修一個、重試一次、再發現下一個缺漏。只要 `missing` 非空,一律回傳失敗,不組出殘缺的提示詞。

**已知會觸發缺漏的情境(非程式錯誤,是資料未到位)**:
- 語氣指引尚未建立(總綱待定案事項:語氣樣本尚未提供,阻塞 P8 階段一,此為已知狀態)。
- Session 的「項目用途」是 DB-05「適用項目」7 個選項之外的值(如日更、個案、實驗)——
  DB-05 目前沒有對應規則類別,屬 schema 落差(另見「項目用途落差說明.md」),此時組稿會正確
  報錯,不會硬套錯的規則。
