## 五項輸入的來源與比對邏輯

| 項目 | 來源 | 比對方式 |
|---|---|---|
| 1. 牌卡資料 | DB-03/04 → DB-02 | 取明細的「抽出順序」字串(權威順序,非關聯欄位),逐一解析後查 DB-02 |
| 2. 對應規則現行版 | DB-05 | 用 Session 的「項目用途」比對 DB-05「適用項目」,篩「狀態=現行」 |
| 3. 月主題包 | DB-08 | 用使用者指定的月份(YYYY-MM)直接比對 DB-08「月份」。**v1.5:項目用途=大眾占卜時此項選填**,缺之不報錯,提示詞省略該區塊(總綱第四章:一更大眾占卜獨立於月主題) |
| 4. 語氣指引現行版 | DB-14 | **v1.5 讀取規則**:篩「標題」以「語氣指引」開頭、「核可狀態」=已核可,解析標題內版本號 vX 取最大值。若該筆「連結」有值,視為指標,依連結取回該 Notion 頁面全文取代「內容」欄 |
| 5. 輸出格式 | DB-05(同第 2 項規則頁的「輸出格式」欄) | 與第 2 項同一次查詢取得 |

五項共用的解析邏輯集中在 `lib/generate/sections.ts`(`resolveCardsSection`/`resolveRule`/
`resolveTheme`/`resolveToneGuide`),單篇組稿(`compose.ts`)與批次組稿共用同一套函式,
不重寫兩份。

## 明細挑選 / 組稿模式(對應 SOP「逐日生成」與 v1.5 批次組稿)

- Session 模式=單筆:只有一筆明細,自動取用。
- Session 模式=批次(如 14 天占卜批次):支援兩種模式,使用者可在組稿前手動切換,兩種都要有:
  - **模式甲・整批一稿**:組出一份提示詞,含全部明細的日期+牌卡資料,規則/月主題包/語氣指引/
    輸出格式各一份不重複,結尾加註「請依上列 N 個日期分別產出 N 篇」。
  - **模式乙・逐篇分稿**:每筆明細各自組一份完整提示詞,清單呈現,各自獨立複製。
  - **模式預判的實作簡化**:v1.5 原文說「App 依 Session 結構預判(全明細同規則→預設甲;
    否則預設乙)」。但目前 schema 下,同一 Session 的所有明細一律用同一條 DB-05 規則
    (規則是依 Session 的「項目用途」查,DB-04 明細本身沒有可覆寫的規則欄位)——換句話說,
    「明細各用不同規則」這個模式乙適用情境在目前資料模型下不會發生。因此目前預設一律為
    模式甲,使用者仍可手動切換模式乙(滿足「不得只實作其中一種」)。若未來 DB-04 新增
    per-明細規則欄位,才需要真正的結構預判邏輯,屬 schema 變更,需另外提案。

## 批次組稿的速率限制與進度顯示

批次的牌卡資料**逐筆**呼叫 `/api/generate/detail-cards`(前端依序 await,不並發),
一方面讓既有的節流佇列(`lib/notion/client.ts`)自然分散請求,另一方面讓使用者在畫面上
看到「已完成 N/M 筆」的即時進度,而不是送出一個大請求悶著等。共用的規則/月主題包/語氣指引
只在 `/api/generate/context` 呼叫一次,不隨明細筆數重複查詢。

## 部分失敗處理

批次組稿中,若某幾筆明細抽牌資料不完整或牌卡查無資料,該幾筆會被排除在提示詞之外,並在
畫面上明確列出「以下 N 筆未能取得牌卡資料」及原因,不靜默跳過(委派書六、Notion API 限制
處理要求)。

## 逆位記法解析

總綱 v2.3 定案:逆位不新增欄位,以「抽出順序」字串內的識別碼後綴 `R` 表示(如 `TA-05R`)。
`lib/generate/cards.ts` 的 `parseDrawOrder()` 依此解析每個識別碼是否逆位,再用不含 R 的識別碼查 DB-02。

## 語氣指引版本挑選與連結全文

- `lib/generate/toneGuide.ts` 的 `pickLatestToneGuide()`:對候選清單逐一用正規表達式
  `/語氣指引\s*v(\d+)/i` 抓版本號,取數字最大者。標題格式不符(抓不到版本號)的候選會被忽略。
- 若挑到的那一筆「連結」有值(v1.5:內容為指標說明、實際全文在連結指向的頁面),
  `lib/notion/queries.ts` 的 `extractNotionPageId()` 從網址解析出 Notion 頁面 ID
  (支援無 dash 的 32 碼十六進位格式,以及帶 dash 的 UUID 格式),再用
  `fetchNotionPagePlainText()` 遞迴讀取該頁全部區塊的文字內容組成純文字,取代「內容」欄。
  連結解析失敗或取回全文為空,都算組稿缺漏,不會靜默退回使用「內容」欄的殘缺說明文字。

## 缺一即報錯

各區塊解析函式回傳 `{ok:true, value}` 或 `{ok:false, missing:string[]}`,呼叫端把所有缺漏
累積起來一次性回報(不是查到第一個缺漏就中斷),讓使用者一次看到所有卡住的地方。只要有任何
缺漏,一律回傳失敗,不組出殘缺的提示詞。

**已知會觸發缺漏的情境(非程式錯誤,是資料未到位或 schema 落差)**:
- Session 的「項目用途」是 DB-05「適用項目」7 個選項之外的值(如日更、個案、實驗)——
  DB-05 目前沒有對應規則類別,屬 schema 落差(另見「項目用途落差說明.md」),此時組稿會正確
  報錯,不會硬套錯的規則。
